A calm, step-by-step guide for non-developers who want to work with code confidently. No experience required — just follow along.

💻 VS Code 🔧 Git ☁️ GitHub ✨ Claude Code 🤖 GitHub Copilot

1 The Big Picture

Before touching anything, get a mental model of how all the tools fit together. This will make every step easier.

💻 VS Code

Where you write and edit code. It's a free text editor made by Microsoft, built for developers but usable by anyone.

🔧 Git

A tool that tracks every change you make — locally on your computer. Think of it as "undo" on steroids, with a full history.

☁️ GitHub

A website that stores your project online and lets you collaborate with others. Git sends your history here.

✨ Claude Code / Copilot

An AI assistant that lives inside VS Code. It helps you write, understand, and fix code — but doesn't touch Git or GitHub.

Key insight: These tools work together — you edit in VS Code, track history with Git, back up and share via GitHub, and get help from AI along the way. You don't need to master all four at once.

2 Setup Guide

Let's get everything installed. Follow each subsection in order — it should take about 30 minutes.

2.1 Create a GitHub Account

  1. Go to github.com and click Sign up for a free account.
  2. Verify your email address — GitHub will send you a confirmation link.
  3. Choose a username you're happy with. It will be public and visible on all your repositories and contributions.
Tip: Choose a professional username — future employers or collaborators will see it.

2.2 Install VS Code

  1. Go to code.visualstudio.com and download the installer for your operating system (Windows, Mac, or Linux).
  2. Run the installer with the default settings — no special configuration needed.
  3. Open VS Code. Here's what you'll see:

2.3 Install Git

Git is a separate tool from VS Code and GitHub — it must be installed on your computer directly.

  1. Go to git-scm.com/downloads and download the version for your OS.
  2. Windows: Run the installer with default settings. Mac: Run this command in your terminal: 
    xcode-select --install
    Linux (Ubuntu/Debian): 
    sudo apt install git
  3. Verify the installation by opening the VS Code terminal (Ctrl+`) and running: 
    git --version
    You should see something like git version 2.43.0.

2.4 Authenticate GitHub in VS Code

  1. Open the Source Control panel in VS Code — it's the branch icon (⑂) in the Activity Bar on the left, or press Ctrl+Shift+G.
  2. VS Code will prompt you to sign in to GitHub. Click "Sign in with GitHub".
  3. A browser window will open. Log in to your GitHub account and click Authorize Visual Studio Code.
  4. Personal Access Token (PAT): If the terminal ever asks for your GitHub password, that no longer works. Instead, go to GitHub → Settings → Developer Settings → Personal Access Tokens → Generate new token. Paste that token as your "password" in the terminal.
  5. SSH keys are an alternative — more secure and you never need to type a password. Optional, but worth setting up later. See GitHub's SSH guide.
Good news: You usually only need to do this authentication step once per computer.

2.5 Install an AI Assistant

  1. Open VS Code and go to Extensions — press Ctrl+Shift+X (Windows/Linux) or Cmd+Shift+X (Mac).
  2. Search for "Claude" and install the Anthropic Claude extension.
  3. Sign in with your Anthropic account or paste your API key when prompted.
  4. The Claude panel will appear in the sidebar — click it to start a conversation with the AI.
Tip: Claude Code works best for open-ended questions and large edits. Ask it things like "Explain this function" or "Refactor this to be cleaner."
  1. Go to Extensions (Ctrl+Shift+X) and search "GitHub Copilot".
  2. Install both GitHub Copilot and GitHub Copilot Chat.
  3. Sign in with your GitHub account. Copilot requires either a paid subscription ($10/month) or the free tier — check github.com/features/copilot for current pricing.
  4. Once active, Copilot automatically suggests code as you type — shown in grey "ghost text".
Tip: Copilot's autocomplete is great for repetitive code. Use Copilot Chat (Ctrl+Shift+I) for longer conversations.

3 Working with Repos

A "repo" (repository) is where all the files for a project live, along with their full change history.

3.1 What is a Repository?

Think of a repo as a project folder — but one that remembers every version of every file ever saved. You can scroll back through history, undo mistakes, and see exactly who changed what and when.

Every repo has two copies: one on your computer (the "local" copy) and one on GitHub (the "remote" copy). Git keeps them in sync.

3.2 Clone an Existing Repo from GitHub

Cloning = downloading a copy of a project that stays connected to GitHub so you can push and pull changes.

  1. Find a repo on GitHub. Click the green "Code" button → copy the HTTPS URL.
  2. In VS Code, open the Command Palette: Ctrl+Shift+P (Windows/Linux) or Cmd+Shift+P (Mac).
  3. Type "Git: Clone" and press Enter. Paste the URL.
  4. Choose a local folder on your computer to save the project in.
  5. VS Code will download the project and ask if you want to open it — click Open.
Cloning in the terminal: 
git clone https://github.com/username/repo-name.git

3.3 Open an Existing Local Repo

  1. In VS Code: File → Open Folder and navigate to the folder on your computer.
  2. If the folder contains a hidden .git subfolder, VS Code will automatically recognise it as a Git repo.
  3. Open the Source Control panel — it will show any uncommitted changes.

3.4 Create a New Repo

  1. Create a new folder on your computer and open it in VS Code (File → Open Folder).
  2. Open the Source Control panel (⑂) → click "Initialize Repository". This creates the hidden .git folder.
  3. Go to github.com → click "+" in the top right → "New repository". Name it and click Create. Do not add a README (you'll push one from your computer).
  4. Back in VS Code terminal, run the commands GitHub shows you: 
    git remote add origin https://github.com/YOUR_USERNAME/YOUR_REPO.git
git branch -M main
git push -u origin main
  5. Refresh GitHub in your browser — your files should appear.

4 The Daily Workflow

Once set up, this is the loop you'll repeat every time you work on a project.

4.1 Edit Your Files

Open files in VS Code and make your changes. VS Code shows modified files in the Explorer with coloured markers: M (modified, in yellow/orange) and U (untracked — new files Git hasn't seen yet, in green).

4.2 Stage Your Changes

Staging means choosing which changes to include in your next save point. You can stage all files, or pick specific ones.

  • In VS Code: open Source Control panel → click the "+" icon next to each file, or click "Stage All Changes".
  • Via terminal:
git add filename.js     # Stage one file
git add .               # Stage everything

4.3 Commit

A commit is a named snapshot of your staged changes. Think of it as pressing "Save" on your project's entire history.

  • In VS Code: type a short message in the Source Control box → click Commit (or press Ctrl+Enter).
  • Via terminal:
git commit -m "Add login page"
Write good commit messages: Use plain English to describe what you did. "Fix typo in homepage" is much more useful than "update stuff".

4.4 Push to GitHub

Pushing sends your local commits up to GitHub — backing them up and making them visible to collaborators.

  • In VS Code: click Sync Changes or the ↑ push button in the Source Control panel.
  • Via terminal: 
    git push

4.5 Pull the Latest Changes

If others are working on the same project, they may have pushed changes. Pull before you start working to stay in sync.

  • In VS Code: click the ↓ pull button or Sync Changes.
  • Via terminal: 
    git pull
Golden rule: Always pull before you start working. It prevents 90% of merge conflicts.

4.6 Branches

A branch is a separate copy of the project where you can experiment or build a feature without touching the main version. The default branch is called main.

  • In VS Code: click the branch name in the bottom-left corner → "Create new branch".
  • Via terminal:
git checkout -b my-feature-branch

When your work is ready, you merge it back into main via a Pull Request.

4.7 Pull Requests (PRs)

A PR is a proposal to merge your branch into another branch (usually main). It gives collaborators a chance to review your changes before they go live.

  1. Push your branch to GitHub: git push -u origin my-feature-branch.
  2. Go to the repo on GitHub — you'll see a yellow banner saying your branch was recently pushed. Click "Compare & pull request".
  3. Add a clear title and description explaining what you changed and why.
  4. Click "Create pull request". Reviewers can now comment, approve, or request changes.
  5. Once approved, click "Merge pull request" to bring the changes into main.

4.8 Merge Conflicts

A conflict happens when two people edited the same line differently and Git can't automatically decide which version to keep. Don't panic — it looks scary but is fixable.

  1. VS Code will highlight the conflicting sections with coloured blocks and labels: Current Change (yours) and Incoming Change (theirs).
  2. Click one of the options above the conflict: "Accept Current Change", "Accept Incoming Change", or "Accept Both Changes".
  3. After resolving all conflicts, stage the file and make a new commit.
⚠️ If in doubt: Don't just click "Accept Both" reflexively. Read both versions and pick whichever one (or combination) is actually correct.

6 Using AI Assistants

AI assistants are powerful collaborators — but they work best when you understand what they're good at.

6.1 What the AI Does (and Doesn't Do)

AI assistants like Claude Code and GitHub Copilot help you write, explain, and fix code. They live inside VS Code but are entirely separate from Git and GitHub. They don't know about your commits, branches, or push/pull status.

Think of them as a knowledgeable colleague sitting next to you — they can suggest code and explain concepts, but you still decide what to keep, when to save, and when to commit.

6.2 When to Use the AI

SituationUse AI?
Writing new code or a function✅ Yes
Understanding what a piece of code does✅ Yes
Fixing an error message✅ Yes
Writing a commit message✅ Yes, great for this
Deciding when to commit❌ That's your decision
Pushing to GitHub❌ Use VS Code Source Control
Resolving merge conflicts⚠️ AI can help explain, but you must resolve

6.3 How to Use Claude Code

  1. Open the Claude panel in the sidebar (the Claude icon, or via the Extensions panel).
  2. Type your question or request in plain English. Examples: 
    "What does this function do?"
"Write a function that sorts a list alphabetically"
"Why am I getting this error: TypeError: cannot read property..."
"Refactor this code to be cleaner"
  3. Claude will respond in the chat and can insert code directly into your file when you ask it to.

6.4 How to Use GitHub Copilot

  1. Copilot works automatically as you type. When it has a suggestion, it shows grey "ghost text" after your cursor.
  2. Press Tab to accept a suggestion. Press Escape to dismiss it.
  3. For longer conversations, open Copilot Chat with Ctrl+Shift+I and ask questions just like with Claude.
Good habit: Don't commit AI-generated code you don't understand. Ask the AI to explain it line by line first. You're responsible for the code in your repo.

7 Terminal Basics

If you've never typed into a terminal before, this section is for you.

7.1 Don't Panic

The terminal is just a text-based way to give instructions to your computer. There's nothing magic about it — you type a command, press Enter, and the computer responds. That's it.

In VS Code, open the terminal with Ctrl+` (the backtick key, usually top-left of your keyboard) or go to Terminal → New Terminal.

You don't need to memorise these. VS Code lets you do most things with clicks. The terminal is mainly needed for the initial setup and occasionally for Git commands that VS Code doesn't expose directly.

7.2 Essential Commands

CommandWhat it does
pwdShows which folder you're currently in
ls (Mac/Linux) / dir (Windows)Lists files in the current folder
cd foldernameMove into a folder
cd ..Go up one folder level
git --versionCheck that Git is installed
git statusSee what files have changed
git logSee the history of commits (press Q to quit)
clearClear the terminal screen

7.3 Tips for Beginners

  • If you see an error in red, read it carefully — it almost always tells you what went wrong and sometimes how to fix it.
  • Press ↑ arrow to cycle through previous commands — no need to retype.
  • Press Tab to autocomplete file and folder names.
  • If a command seems stuck, press Ctrl+C to cancel it.
  • The terminal always has a "current folder" — make sure you're in the right project folder before running Git commands.

8 Common Mistakes & Fixes

Everyone hits these at some point. Here's how to get unstuck quickly.

1
"I forgot to pull before editing and now there are conflicts"
▼
Pull now — Git will merge the incoming changes with yours. Then resolve any highlighted conflicts in VS Code (use "Accept Current/Incoming Change"), stage the resolved files, and commit. 
git pull
2
"I committed directly to main instead of a branch"
▼
For solo work this is usually fine — it won't break anything. For team projects, create a branch next time. If you haven't pushed yet, you can undo the last commit (keeping your changes) with: 
git reset --soft HEAD~1
If you've already pushed to main, it's best to just move forward rather than trying to rewrite history.
3
"VS Code is asking for my GitHub password in the terminal"
▼
GitHub stopped accepting passwords in 2021. You need a Personal Access Token (PAT) instead. Go to: GitHub → Settings → Developer Settings → Personal Access Tokens → Tokens (classic) → Generate new token. Copy the token and paste it where the terminal asks for your password. It won't show as you type — that's normal.
4
"I accidentally staged a file I didn't mean to commit"
▼
Before committing: In the Source Control panel, right-click the file and choose "Unstage Changes". Or via terminal: 
git restore --staged filename.js
After committing but before pushing: Use git reset --soft HEAD~1 to undo the commit.

After pushing: Create a new commit that removes the file. Avoid trying to rewrite already-pushed history.
5
"My push was rejected"
▼
This almost always means someone else pushed changes to the same branch after your last pull. The fix is simple: pull first, then push. 
git pull
git push
If there are conflicts after the pull, resolve them first (see #1 above), then push.
6
"I don't see the Source Control panel in VS Code"
▼
Click the branch/fork icon (⑂) in the Activity Bar on the left side of VS Code, or use the keyboard shortcut Ctrl+Shift+G (Windows/Linux) / Cmd+Shift+G (Mac). If the panel is empty, your folder may not be a Git repo yet — click "Initialize Repository" to fix that.
7
"The AI wrote code but I don't know if it's correct"
▼
Test it — run it and see if it produces the expected output. Ask the AI to explain it step by step: "Walk me through this code line by line." Don't commit code you don't understand — if it breaks something, you'll have a hard time fixing it without knowing what it's supposed to do.
